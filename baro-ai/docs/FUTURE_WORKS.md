## 🏛️ 아키텍처 개선

### 1. **DDD 계층형 아키텍처 적용** ⭐⭐⭐

현재 혼재된 구조를 명확한 역할과 책임을 갖는 DDD 계층형 아키텍처로 리팩토링합니다.

#### 현재 문제점
- `search`, `recommend` 등 도메인 패키지와 `config`, `event` 등 인프라 패키지가 최상위 레벨에 혼재
- 패키지 구조가 일관되지 않아 전체 구조 파악이 어려움
- 계층 간 의존성 규칙이 명확하지 않음

#### 목표 구조
```text
src/main/java/com/barofarm/ai/
├── domain/          # 🏛️ 도메인 계층 (비즈니스 규칙)
├── application/     # 🚀 애플리케이션 계층 (유스케이스)
├── infrastructure/  # 🔧 인프라스트럭처 계층 (외부 연동 기술)
└── common/          # 🛠️ 공통 계층 (공용 유틸리티)
```

#### 구현 태스크
- [ ] `domain`, `application`, `infrastructure` 최상위 패키지 생성
- [ ] `presentation`, `*Controller` → `infrastructure/web`으로 이동
- [ ] `*Repository` 구현체, `search/infrastructure` → `infrastructure/persistence`로 이동
- [ ] `event`, `consumer` → `infrastructure/messaging`으로 이동
- [ ] `embedding`, 외부 API 호출 → `infrastructure/external`로 이동
- [ ] 기존 서비스 로직을 `application`과 `domain` 계층으로 분리 및 재배치
- [ ] 계층 간 의존성 규칙(`Infra` → `App` → `Domain`) 검증

### 2. **헥사고날 아키텍처로 전환 (선택적 장기 과제)** ⭐

계층형 아키텍처가 안정화된 후, 필요에 따라 포트(Port)와 어댑터(Adapter) 패턴을 도입하여 완전한 헥사고날 아키텍처로 전환을 고려할 수 있습니다.

#### 전환 목표
- `application` 계층에 `inbound/outbound` 포트(인터페이스) 정의
- `infrastructure` 계층의 각 구현체를 어댑터로 명확히 정의
- 프레임워크로부터의 완전한 독립성 확보

### 3. **DDD 패턴 심화 적용** ⭐⭐

도메인 주도 설계 패턴을 더 엄격하게 적용합니다.

#### 추가 도메인
- [ ] `domain/user/` - 사용자 프로필, 선호도 관리
- [ ] `domain/product/` - 상품 카탈로그, 카테고리
- [ ] `domain/recommendation/` - 추천 엔진, 평가 메트릭
- [ ] `domain/search/` - 검색 쿼리, 결과 랭킹

#### 도메인 서비스
- [ ] `RecommendationDomainService` - 추천 알고리즘
- [ ] `SearchDomainService` - 검색 로직
- [ ] `UserPreferenceService` - 사용자 취향 분석

### 3. **CQRS 패턴 적용** ⭐⭐

읽기/쓰기 모델 분리로 성능 최적화

#### 구현 계획
- [ ] Command 모델 (쓰기): 이벤트 처리 및 상태 변경
- [ ] Query 모델 (읽기): 추천/검색 결과 조회
- [ ] 이벤트 소싱: 상태 변경 이력 관리
